## 1. 產品概述 (Product Overview)
### 1.1 產品願景
提供一個直覺、快速且高度可靠的個人財務紀錄工具，讓使用者能透過行動裝置隨手記錄收支，並透過雲端同步與資料導出功能，確保帳務資料的永續性與可分析性。

### 1.2 核心價值
*   **低摩擦力紀錄：** 最小化使用者輸入時間。
*   **資料安全性：** 確保收支敏感資料在同步與導出過程中的完整性與隱私。
*   **高可靠性：** 嚴謹處理離線狀態與資料衝突。

---

## 2. 功能模組拆解 (Functional Modules)

### 2.1 用戶帳號模組 (User Authentication)
*   **註冊/登入：** 支援 Email/密碼登入。
*   **第三方登入（規劃中）：** Google/Apple ID 登入，用於簡化雲端同步流程。
*   **密碼找回：** 透過 Email 進行密碼重設。
*   **帳號狀態管理：** 紀錄登入 Token 與 Session，支援自動登入機制。

### 2.2 收支紀錄模組 (Transaction Management)
*   **新增紀錄：** 支援「收入」與「支出」兩種類型切換。
*   **分類管理：** 提供預設分類（如：飲食、交通、薪資、娛樂），並允許使用者自定義分類與圖示。
*   **日期選擇：** 預設為「今日」，允許使用者回溯紀錄過去日期。
*   **筆記功能：** 允許使用者為每筆交易添加簡短文字說明。

### 2.3 統計儀表板 (Dashboard & Analytics)
*   **總覽摘要：** 顯示當月總支出、總收入、總餘額。
*   **趨勢圖表：** 提供每月收支趨勢圖（折線圖）與分類佔比圖（圓餅圖）。
*   **篩選功能：** 可依據月份、年份或分類進行資料過濾。

### 2.4 資料同步與備份 (Data Sync & Backup)
*   **自動同步：** 當 App 偵測到網路連線且使用者操作紀錄時，自動將本地資料與雲端資料庫同步。
*   **衝突處理策略：** 若本地與雲端資料時間戳 (Timestamp) 衝突，以「最後修改時間 (Last Write Wins)」為準，並保留衝突紀錄備份。

### 2.5 資料導出模組 (Data Export)
*   **CSV 導出：** 使用者可選擇「特定月份」或「所有紀錄」，將資料匯出為 `.csv` 檔案。
*   **檔案管理：** 支援透過系統分享功能（Share Sheet）將 CSV 傳送至 Email、LINE 或儲存至手機檔案。

## 3. 輸入欄位驗證規則 (Input Validation Rules)

為了確保資料一致性，所有輸入必須符合以下規範：

| 欄位名稱 | 類型 | 是否必填 | 驗證規則 (Validation Rules) | 錯誤提示訊息 |
| :--- | :--- | :--- | :--- | :--- |
| **金額 (Amount)** | Double | 是 | 必須為大於 0 的數字；不接受負數（透過收支類型區分） | 請輸入有效金額 |
| **交易類型 (Type)** | Enum | 是 | 僅限 `Income` 或 `Expense` | 欄位異常 |
| **日期 (Date)** | Date | 是 | 格式為 `YYYY-MM-DD`；不可超過系統當前日期 | 日期不可為未來日期 |
| **分類 (Category)** | String | 是 | 必須屬於系統內已定義的分類 ID | 請選擇分類 |
| **備註 (Note)** | String | 否 | 最大長度 100 個字元 | 已達字數上限 |
| **Email** | String | 是 | 必須符合 Email 標準 Regex 格式 | 請輸入正確的 Email 格式 |

## 4. 異常流程處理 (Exception & Edge Case Handling)

作為資深 PM，我強調「錯誤處理」是決定產品品質的關鍵。

### 4.1 網路連線異常 (Network Issues)
*   **情境 A：紀錄時無網路：** 系統應允許使用者繼續操作，資料暫存在本地資料庫 (SQLite/Hive)，並在 UI 狀態列顯示「目前離線，紀錄將稍後同步」。
*   **情境 B：同步過程中斷：** 必須實作「斷點續傳」與「重試機制 (Retry Mechanism)」。若連續失敗 3 次，則停止自動同步，並在 UI 提示「同步失敗，請點擊重試」。

### 4.2 資料衝突處理 (Data Conflict)
*   **情境：** 使用者在手機 A 編輯了 1/1 的飲食支出，隨即在手機 B 也編輯了同一筆紀錄。
*   **處理：** 後端需檢查 `updated_at` 時間戳。若兩者不一致，以較新的時間戳為主。系統需在後端日誌中記錄衝突事件，以備日後 Debug。

### 4.3 輸入錯誤與空值 (Input & Null Handling)
*   **空值處理：** 若使用者未填寫「備註」，資料庫存儲 `null` 或空字串，前端顯示時預設顯示「無備註」。
*   **非法格式：** 若使用者嘗試在金額欄位輸入非數字字元，按鍵應直接失效或即時跳出錯誤提示，禁止提交。

### 4.4 檔案導出錯誤 (Export Failures)
*   **儲存空間不足：** 若手機空間不足導致無法儲存 CSV，應跳出系統級警告：「手機空間不足，無法導出檔案」。
*   **資料過大：** 若使用者嘗試一次導出十年的資料，應限制最大導出範圍（例如單次最多導出 12 個月），避免 App 發生 OOM (Out of Memory) 崩潰。

## 5. 資料隱私與安全 (Data Privacy & Security)

*   **傳輸加密：** 所有與雲端溝通的 API 必須強制使用 **HTTPS (TLS 1.2+)**。
*   **儲存加密：** 本地 SQLite 資料庫若儲存敏感資訊（如帳號），建議進行加密處理。
*   **資料最小化原則：** 雲端資料庫僅儲存必要的財務紀錄，不收集使用者的聯絡人、通訊錄或精確 GPS 定位。
*   **CSV 安全：** 導出的 CSV 僅存在於使用者的本地檔案系統中，App 不會將導出的檔案上傳至任何第三方伺服器。
